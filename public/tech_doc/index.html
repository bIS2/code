Basic Design of bIS

The Begleitendes Informationssystem (bIS) is a tailor-made system comissionend by Swiss libraries to better solve the deduplication of library holdings (www.speicherbibliothek.ch). It serves specially to deduplicate periodicals in different libraries in Switzerland.
T
his is a description of how bIS work. It aids troubleshooting by providing the troubleshooter with a mental model of how the system is supposed to work. The troubleshooter can then more easily identify discrepancies, to aid diagnosis of problem.

Object id Definition
TA library system (bib) + system number (sys) A title records defined by a MARC record
HOL library system (bib) + system number (sys) A holding from a MARC record
HOS library system (bib) + system number (sys) + holding group number (g) Holdings groups by similarity of its TA

The basic item in bIS is the HOL. Is alway contains (redunantly) data from TA, because TA is not represented in bIS.

In an offline process these HOL have been compared by it’s title information (see programs) to build the Holdings Sets (HOS). They represent all holdings found of a unique title. This list of HOL (hol_out) is transferred to the bIS.

As the comparation by similarity cannot be a perfect process, the HOS must be verified by librarian. They can add or remove HOL and evntually confirm the correctness of a HOS. Every HOS must contain a “owner” (AB)

After this step only HOL are regarded until the final steps.

Roles

Every User can have one role. Exception: A BIBUSER can have also a RESUSER role.

bIS Functions explained

In the following chapter are described solutions for some essential functions of the bIS. Further details can be found in the program files.

Pre-processing of MARC data

The MARC data for the bIS has been prepared in an offline process. This is for historical reasons, as the comparation of titles has been requested first as a one-time work. This step contains:

  • reading MARC data into a database on a subfield basis
  • statistical analysis to learn about the heterogeneity of the data and to determine the best way to compare TA
  • recognition of the content of field 866a

Most pre-processes have later been integrated into bIS (editing of 866a, recalculation of HOS).

The result has been tranferred to the bIS by these tables:

Table Content Use
hol_out all HOL (grouped to HOS by sys1) Read into TABLE holdings and TABLE holdingssets
tit_freq list of frequently appearing string in f245a Copied to TABLE tit_frequ
hol_values Knowledge base for f866a Copied to TABLE hol_values
libraries List of libraries Read into TABLE libraries

Workflow Manager

For a complete description and further details of the workflow consult the User Manual (german).

The workflow of bIS are managed through the states of HOL and HOS. User action can change the state. According to the state the HOL and HOS are presented to other roles for further treatment defined in the workflow scheme.

Calculation of Holding pattern from hol_nrm

The hol_nrm consists of 1..n pattern representing the normalized form of a HOP:

HOP_start hyphen HOP_end
VVVVvvvvYYYYyyyy-VVVVvvvvYYYYyyyy

The completeness of data in the pattern depends on the HOP information in 866a.

Pattern Explanation
VVVV Volume
vvvv Volume extension
YYYY Year
yyyy Year extension
hyphen if HOP_end is empty: a non terminated HOP
no hyphen HOP is not extended (normally 1 year)

Calculation of HOL pattern

Similarity Search

HOS are sets of HOL that represents the same periodical. As title information is not always written in the same way, the system has to find a way to recognize titles with similar content. This is archieved with different methods:

Method Explanation Examples
“Normalize field content Make similar content more equal (p.e. change all german umlaut to ae oe ue; put all text to lowercase; trim values) Aerger/Ärger => aerger
Standardize values Recognize values and replace it 2-wöchentlich/2-wtl./zweiwöchentlich/2wöchentl. => 2w
Compare HOL with unreliable data Use a similarity (postresql) for weighted comparison Proximity of HOL are defined by a score between 0 and 1

These steps have been done offline with standard tools. The latter tool has been implemented in the bIS als Option 1b.

HOS confirmation

All HOS that only contains 1 HOL, are confirmed at the beginning. But when changing the confirmation, this rule not applies any more.

holdingssets.autoconfirm = 1

HOS confirmation puts all containing HOL into the state “HOL correct”.

holdings.state = “correct”

Occurrence Pattern

Calculation: The occurrence pattern in always calculated for an entire HOS. If changes are made within a HOS, the pattern has to be recalculated. For this the normalized field f866a called hol_nrm is used. All segments (separated by ;) of all HOL are used to create a segment pattern.

Every segment is represented by a block

  • Cut all segments
  • Exclude void
  • Put in array ordered by starting “JJJJ” and starting “VVVV”
  • Compare for every HOL the occurrence. Put 1 otherwise 0
  • Give a weight to every segment generated by number of years and volumes
  • Check occurrence for every HOL
  • Check for completeness “j”
  • Changing the mechanism: holdings.reserved = 1 ; holdings.force_xxx = 1 ; holdings.is_pref = 1
Color Explanation
Red The HOL with a biggest weight or mayor number of occurrecnes or with the field holdings.force_owner = 1 will be owner = red.
In the case of ambiguity the HOL with the field holdings.is_pref = 1 will be choosed. (meaning, that the library ###)
Yellow Create the complement of the pattern and the owner pattern. There is no selection of preference.
Blue All others
White HOL without normalized pattern in holdings.

“Forcing”: Forcing is the action to change the automatically calculated HOS.

Action Explanation
force_owner Recalculation of HOS
force_aux The segments choosen by the user are put yellow.
holdings.aux_ptrn: Puts “1” to this segments.
holdings.is_aux = true
force_blue All other yellow segments are put blue.
holdings.aux_ptrn and .owner_ptrn: Puts “0” to this segments.
holdings.is_aux = false.
holdings.is_owner = false

Recalculation: The HOS is recalculated after following actions:

  • Eliminating a HOL
  • After editing 866a
  • Adding a HOL
  • Divide a a HOL
  • Reserve a HOL
  • When the user forces recalculation

- Backup must be stored on an external server
